# 教训库

> 不是记 bug 修了什么——是记那次犯错时，脑子里在走什么错误路径。
> 每次犯错追加一条。格式：现象 → 思维路径 → 为什么错 → 修正规则。

---

## 审查类

### #1 审查只看函数体，不看调用方

**日期**: 2026-06-04
**现象**: `_should_idle_rest` 函数签名改为 `(player, needs)`，函数体审查通过。但 `tick()` 还在传 `eval_need`（字符串），F5 崩了——玩家永远不 idle。
**当时的思维路径**: "函数逻辑看起来没问题 → 标签用了 → 数据表对 → 过"。把文件当独立对象审查，不检查谁在调用这个函数。
**为什么错**: GDScript 不做类型检查，错参不会编译报错。审查函数的终点不是函数体——是每个调用方。
**修正**: §3.0 铁律一：签名变更 → grep 全库调用方，逐个验证参数匹配。

### #2 审查不看边界条件

**日期**: 2026-06-04
**现象**: 自主值 0↔100 震荡、`craft_spear` 闪现消失。都是边界 flicker——火安全区半径 3 vs 狼威胁半径 3 角格不同，扫描半径 6 vs 玩家移动 1 格进出。
**当时的思维路径**: 验了"正常情况"——自主 100、胜任 50、关联 90。数值看起来对，审完就走。
**为什么错**: 玩家永远在边界上玩。扫描半径边缘、安全区边界、首 tick 空值——这些才是 bug 的栖息地。"正常值"永远不会出问题。
**修正**: §3.0 铁律二：每次审查至少模拟一个边界条件。

### #3 看最终代码，不看变更

**日期**: 2026-06-04
**现象**: 审查人卡大脑重构代码——标签表对了、函数对了、数据流对了。但看不出来 `_should_idle_rest` 签名变了，也看不出 `eval_need` 那一行是旧系统残留。
**当时的思维路径**: 读代码就像读书——从头到尾，理解逻辑，判断对错。
**为什么错**: 重构代码的"正确性"不在它本身——在它和旧代码的差分里。最终代码看起来很自洽，因为它已经是一个整体了。只有 diff 能暴露：这一行是新的还是旧的？这个参数为什么和之前不一样？这个模块接线还在吗？
**修正**: §3.0 铁律三：审查时看 diff。对每行改动问"这是改了什么、它连着什么"。

## 设计/架构类

### #4 handoff 写"应该怎样"，不写"改什么"

**日期**: 2026-06-02
**现象**: UI 容纳需求的 handoff 写了"如果卡牌有容纳，点击就能看到""如果卡、格子有设定就应该展示"——全是语义描述。Cursor 实现了代码但视觉上不可见，角标 11px 小字贴在棕色背景上，用户完全看不到。
**当时的思维路径**: "说清楚需求就行了，Cursor 会理解的"。
**为什么错**: Cursor 不是设计师。它需要"改 `build_cell_lines` 的 overlay 标题逻辑 => `title = overlay.def.display_name`"，不是"格子有窝应该显示窝名"。语义越精确，实现离预期越远。
**修正**: PROTOCOL §6 模板：每条改动 = 文件路径 + 函数名 + 现状 + 改为。不写"应该""期望"。

### #5 新机制不加"可视化清单"

**日期**: 2026-06-02
**现象**: den 建成了、humus 产生了、窝里有动物——全部存在，玩家看不到。因为没有地面标记、没有信息栏覆盖、没有选中提示。
**当时的思维路径**: "后端逻辑对了就行，UI 是属于另一个 handoff 的事"。
**为什么错**: 玩家不读代码。后端机制的存在对玩家而言唯一的证据是视觉呈现。没有可视化 = 这个机制不存在。
**修正**: PROTOCOL §3.1b：任何新卡新机制，handoff 必须写外观 UI（地图上看得到）+ 选中 UI（点上去查得到）。审查时对照清单逐条过。

### #6 标签驱动化不写死约束，Cursor 就填硬编码

**日期**: 2026-06-04
**现象**: 重构人卡大脑，handoff 给了五层架构、数据表、代码模板。但没在第一行写"每行决策必须从标签推导"。结果 Cursor 把骨架填了标签表，把血肉填了魔法数字（`有狼 → 减 80`、`没火 → 减 60`、`if "spear" in tools`）。
**当时的思维路径**: "架构图给了、函数签名给了、示例代码给了——约束已经隐含在结构里了"。
**为什么错**: 对 Cursor 来说，`if _has_weapon(player)` 和 `if _PlayerBrainTags.has(player, "has_weapon")` 都是合法写法，选哪个取决于谁更短更容易写。架构上的标签化是一个哲学选择，不是一个技术约束。不写死，就会被绕过。
**修正**: 任何架构级 handoff 第一行必须写铁律——"所有决策从标签推导，零硬编码条件，禁用任意魔法数字"。不是建议，是硬约束。

### #7 相信"自己看得懂的数据"就够了

**日期**: 2026-06-04
**现象**: 审查 session_report §8 大脑数据——自主 100、胜任 50、关联 90。数值正常，审完通过。但没发现"狼显示搬肉但实际没叼肉卡 1213 tick""玩家在找食和造矛之间思维切换"——这些在数据里是"取肉""状态是 forager 和 craft_spear 交替"。
**当时的思维路径**: "数值稳定、无异常信号 → 通过"。
**为什么错**: 数值只是大脑的体检报告。行为线（状态连续切换、意图没有完成就放弃）才是真正的症状。只读数不读行为链 = 看了体温没看病人。
**修正**: 审查 session_report 时不止看 §8 数字，必须读 §2 的行为线——连续 tick 状态序列、不动时长、意图-可供性-状态的因果链匹配。

## 流程/沟通类

### #8 手把手给代码模板，导致审查跳过集成

**日期**: 2026-06-04
**现象**: handoff 给了精确的 AFFORDANCE_TABLE、NEED_TAG_RULES 代码模板。Cursor 照模板实现，代码本身正确。但 `tick()` 里 `_should_idle_rest` 调用没更新——模板没覆盖集成层。
**当时的思维路径**: "模板写了每一行的样子，Cursor 照着填，填出来审查通过就行"。
**为什么错**: 代码模板天然聚焦在"怎么写函数"上，不聚焦在"函数怎么连在一起"。给了模板反而让我放松了集成检查——因为"模板覆盖了所有东西"的错觉。
**修正**: handoff 中代码模板的最后一行必须写"所有调用方同步验证"。审查时追踪从 `tick()` → `evaluate()` → `_should_idle_rest()` → `action_runner._tick_idle_scan()` 的完整调用链。

### #9 不要给设计缺陷打代码补丁

**日期**: 2026-06-04
**现象**: 狼叼肉回窝，兔子堵在窝门口不逃，狼进不去窝。于是加了"放下肉→驱赶兔子→再捡肉"的逻辑。之后出现 carry 标记竞态：狼以为自己还在搬肉，实际肉已放下。狼困在窝前乱晃，0 猎杀 0 肉，不动 1213 tick。
**当时的思维路径**: "规则写好了：狼必须搬着肉进入窝格才算交付。现在兔子堵门→进不去→那就先驱赶再进来。加个逻辑补一下就行。"
**为什么错**: 
1. 狼不需要"驱赶"。那是狗的行为。自然里狼靠近到一定范围，猎物自动逃跑。这个 bug 的根因不是"狼进不去"，是"兔子看到狼居然没逃"。
2. 窄门困住了自己。"必须叼肉踏入窝格"是硬门槛。一旦任何东西挡在格子上，整个流程僵死。然后补丁（放下驱赶）又产生了新 bug（carry 标记竞态）。补丁套补丁。
3. 把动物习性写反了。狼是自然生态的一部分——猎物见到就会跑。不是执行"我要先进窝所以得先清场"的任务脚本。
**修正**: 
- 设计的一条纪律：**如果一个机制需要给自然行为加"驱赶"类补丁，说明自然行为本身没写对。**
- 修法：反向追溯——不是狼怎么进门，是兔子为什么没逃。逃走在先，驱赶是狗。
- 窄门拆掉：不要求精确同格交付，幼狼出窝接肉。
