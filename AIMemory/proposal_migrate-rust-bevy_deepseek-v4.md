---

## 背景

当前项目用 Godot 4.6.1 + GDScript。生态二期全量落地后 L0 993 断言 PASS。

**核心问题**：GDScript 性能天花板。Profiler 实测 _game_tick 642-805ms/次（目标 < 16ms）。经过三期优化（SpatialIndex + 改桶 + 寻路缓存），降到 717ms。瓶颈不是算法，是语言——`card_has_tag` 在 GDScript 里一次 15ms，C++/Rust 里是纳秒。

**设计方向**：游戏最终复杂度是当前的 100-1000 倍。标签驱动 → WorldRules 计算关系 → 涌现。性能必须和卡量不成正比。

---

## 提案

迁移到 **Rust + Bevy 引擎**。

### 为什么 Rust
- 编译期把 `card_has_tag` 变成 bitwise AND（纳秒），不是运行时字符串比较（毫秒）
- 零开销抽象：标签用 enum，规则用 trait，和我们的"标签即因果"本体一致
- 编译器是全球最严格的自动化检错

### 为什么 Bevy
- Rust 生态最成熟的 ECS 框架。ECS 的 Component = 我们的标签
- 原生 Observer 事件系统：卡状态变化 → 自动触发规则 → 不用轮询
- 2D 渲染、UI（CSS-flexbox）、精灵动画全部内置
- 多个已上架 Steam 的商业游戏（Tiny Glade 100 万愿望单、Tunnet 等）

### 架构变化

```
现在（GDScript，轮询式）：
  每帧 for card in GameState.cards:
    card_has_tag(card, "predator")? → 查字典 → 15ms
    → 再扫一遍找 prey → ...

迁移后（Rust/Bevy，事件驱动）：
  卡移到新位置
    → Bevy Observer 自动触发
    → 只评估这张卡涉及的规则（纳秒）
    → 产生交互
```

### 开发工作流不变

你说设计 → deepseek-v4 拆 handoff → Cursor 实现 Rust 代码 → 你 F5 验收。

区别：
- 改代码到看到效果多 10-30 秒编译
- 视觉调试（调位置）从拖拽变改数字
- 日常新卡迭代：加一个 `.ron` 定义文件 + 一个 PNG，**零代码**

### 1. Cursor 对 Rust + Bevy 的驾驭能力？

**能写，但有边界。**

| 能力 | 判断 |
|------|------|
| Bevy ECS（Component / System / Resource / Query） | ✅ 能写，是 Cursor 最擅长的迁移形态 |
| Observer / Event（Bevy 0.14+ `Trigger` / `Observer`） | ✅ 能写，需锁定 Bevy 版本，API 迭代快 |
| bevy_ui（侧栏、按钮、布局） | ⚠️ 能写，但远不如 Godot Control 顺手；复杂 UI 迭代慢 |
| 2D 精灵 / 帧动画 / 网格地图 | ✅ 能写；tilemap 需第三方 crate 或自研格网（与现项目格网一致，反而好迁） |
| 编译报错（borrow checker / trait bound） | ✅ 多数能独立修；大型 ECS 交叉引用偶需人工拆系统 |
| `.ron` 数据驱动、零代码加卡 | ✅ 能设计；但「零代码」只覆盖**数据层**，规则仍要 Rust trait 实现 |

**实话**：Cursor 写 Rust 的质量通常**高于** GDScript（类型系统帮大忙），但**反馈回路变慢**——改一行 → `cargo build` 10–60 秒 → 跑测试。Godot 的 F5 即时性会丢失，除非搭 `bevy` + `cargo watch` + headless 测试脚本。

**对提案里「card_has_tag 变 bitwise AND」的修正**：方向对，但表述过简。Rust 里更自然的是 **ECS Component 存在性查询**（`Query<&Predator>`），不是给每张卡做字符串 tag 比较。性能提升主要来自：(1) 无 GC；(2) 无动态字典；(3) 编译期布局。不是魔法，是架构匹配。

| L0 断言 | 993 条，背后是完整生态 + 营地 + 贸易 + 玩家脑 |

**人天估算（1 名熟悉 Bevy 的 Rust 开发者 + Cursor 辅助）：**

### 3. 不可行风险：有没有会卡死的技术障碍？

**没有「绝对不可行」，但有高风险区。**

| 风险 | 严重度 | 说明 |
|------|--------|------|
| **Bevy 版本碎片化** | 高 | 0.13→0.14→0.15 API 变动大；需锁定版本 + 升级预算 |
| **UI 重建成本** | 高 | 侧栏、拖卡、选中高亮、容纳列表——Godot 已打磨；Bevy UI 要从零做，非性能问题，是**工期问题** |
| **玩家脑 / 意向系统** | 高 | `player_needs_manager` 单文件 1400+ 行；Rust 可写，但迁移量 = 半个 POC |
| **Observer 万能论** | 中 | 提案「移动触发规则、不轮询」**只覆盖部分逻辑**。繁殖 tick、天气、腐解计时、饥饿衰减仍需 **Scheduled System**。Observer 减轮询，不能消灭 tick |
| **2D tilemap** | 低 | Bevy 核心无官方 tilemap；用 `bevy_ecs_tilemap` 或自研格网（本项目已是格网，反而简单） |
| **精灵帧动画** | 低 | `TextureAtlas` + `Animation` 成熟 |
| **工具链** | 中 | 团队需装 Rust toolchain；非技术负责人无法 F5 验收，需编译产物或 CI 跑测 |
| **「语言瓶颈」论断** | 中 | Phase 2 后 Profiler 显示真正热点是 **`defines_domain` / `in_camp_home_territory` / `best_feed_source`**（各 ~60s 累计），不全是 `card_has_tag`。Rust 会快，但**架构/cache 不做，Rust 也会慢** |

**不会卡死，但可能「迁到一半进退两难」**：Godot 线停更 3 个月，Bevy 线 UI 未就绪，两头落空。这是最大项目风险，不是技术风险。

### 4. 首选方案：全量迁（A）还是 POC（B）？

**强烈推荐 B：先做技术 POC，再决策。**

| | A 全量迁移 | B POC 先行 |
|--|-----------|-----------|
| 风险 | 极高：一次性抛弃 993 L0 + 已打磨 UI | 可控：Godot 线继续，Bevy 线验证假设 |
| 验证核心假设 | 迁完才知道 Rust 是否真 <16ms | **2–3 周**可测 tick 耗时 |
| 回退 | 几乎不可能 | 随时放弃，损失有限 |

**建议 POC 范围（最小可验证）：**

```
bevy 0.14 + 36×24 格网
├── CardDef 从 .ron 加载（草、羊、狼 3 种）
├── SpatialIndex（port 现有逻辑）
├── 每 tick：羊找草 → 移动 → 吃；狼找羊 → 寻路 → 猎
├── headless 跑 1000 tick，打印 tick 耗时
└── 20 条从 L0 精选的断言（Rust 内嵌测试）
```

**POC 通过标准（决策门）：**

1. 100 张卡、100 tick 平均 **< 5ms**（不是 16ms，留余量）
2. Cursor 能在 1 次会话内改规则并编译通过（验证开发流）
3. 你（非技术负责人）能双击 exe 看到羊狼跑起来

**若 POC 通过** → 分期迁移：先模拟核（Rust）+ Godot 壳（FFI）或纯 Bevy，按模块切（生态 → 玩家 → UI）。  
**若 POC 未达 10× 加速** → 继续在 Godot 做 Phase 3（域缓存 + type tag 预建），成本远低于迁移。

**不推荐 A 的理由一句话**：当前 Godot 项目不是原型，是**有 993 断言的生产级模拟器**；全量翻写等于重写整个产品，而性能问题可能用 Phase 3 架构优化解决 80%。

### 5. 综合结论

| 问题 | 简短回答 |
|------|----------|
| 能写 Rust + Bevy 吗？ | **能**，ECS 部分质量高；UI/工作流是短板 |
| 多久？ | POC **3–4 周**；生态地基 **3–4 月**；全量对齐 **6–12 月** |
| 技术障碍？ | 无死穴；**UI 重建 + 双轨停滞** 是最大项目风险 |
| A 还是 B？ | **坚决 B** |

**给 deepseek-v4 的补充**：迁移解决的是「语言上限」，不自动解决「架构轮询」。即使迁 Rust，`defines_domain` 927 次/10s 这种逻辑仍需缓存设计。建议 POC 与 Godot Phase 3 **并行立项**，用数据决定赌哪边。
