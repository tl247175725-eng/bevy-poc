# AIMemory 协作协议 — 方寸商国：桃花源记

> 本文件是 DeepSeek（策划助手）与 Cursor（主程序）之间的协作约定。双方在每轮工作开始时读取本文件，结束时按约定写回。

---

## 1. 角色

| 角色 | Agent | 能力 | 职责 |
|------|-------|------|------|
| 策划/决策 | 用户 | 设计方向、创意、逻辑分析 | 审核方案、确认方向、F5 验收 |
| 策划助手 | DeepSeek | 读代码、写文档、架构审计 | 设计分析、数值设计、handoff 撰写、**Cursor 合入后审查代码** |
| 主程序 | Cursor | 写代码、跑终端、跑测试 | 按 handoff 实现、修 bug、记 fix-log |

## 2. 核心流程

### 2.1 常规任务（新卡/参数/bug）

```
用户+DeepSeek 讨论 → DeepSeek 写 handoff → 用户确认 → Cursor 实现 → DeepSeek 审查 → 用户 F5
```

### 2.2 架构改造（调度层/标签系统/管线重构）

```
第一步：DeepSeek 写"架构命题"（描述目标状态 + 断点位置 + 方向建议，不写具体改法）
        ↓
第二步：Cursor 做架构审查（读命题 + 代码 → 回复可行性、改动范围、遗漏风险）
        ↓
第三步：对齐后 Cursor 拆执行步骤（自己规划顺序和粒度，DeepSeek 不再写逐行 handoff）
        ↓
第四步：DeepSeek 最终审查（标签合规 + 设计一致性 + 管线完整）
        ↓
用户 F5 验收
```

**架构命题不是执行单**。它只定义"最终应该是什么样"和"为什么现在不是这样"。Cursor 是架构参与者——它来验证、反驳、补充方案，然后自己拆 handoff。

### 2.3 审查触发

DeepSeek 在看到 work.log 中出现 Cursor 的 WORK_END 后，主动启动审计，不等用户发指令。用户只需 F5 验收。

**审查触发规则**：DeepSeek 在看到 work.log 中出现 Cursor 的 WORK_END 后，主动启动审计，不等用户发指令。用户只需 F5 验收。

## 3. 审查标准（DeepSeek 执行）

Cursor 回复"完成"后，DeepSeek 检查以下：

### 3.0 审查纪律（每次必执行）

基于反复犯错的根因分析，三条铁律：

1. **函数签名变更 → grep 所有调用方，逐个验证参数匹配。** 审查不能只看改动的函数体。函数名、参数列表、返回值变了——必须搜索整个代码库，确认每个调用方都同步更新。这是 `_should_idle_rest` 漏查的根因。

2. **边界条件必须模拟。** 半径边界（扫描 6 格→第 7 格）、首 tick 空值（大脑未初始化）、状态临界点（全需求 100→idle 是否触发）。只验证"正常情况下"的数据是假的——玩家永远在边界上玩。

3. **审查时看 diff，不是只看最终文件。** 对每行改动问"这是改了什么、它连着什么"。函数签名变动、模块接线、旧系统依赖残留——这些东西在最终代码里不显眼，在 diff 里一眼暴露。

### 3.1 设计方法：真实世界抽象

**原则**：我们是在用抽象逻辑（标签 + 能力）模拟真实的复杂系统。不是用复杂系统模拟复杂系统，而是用两个系统一个维度去抽象真实世界——只保留"卡"这种抽象封装单位，让内在真实保证自洽。

**操作**：设计任何新卡、新行为、新关系时，回答三个问题：

1. **它在真实世界里怎么活？** 食性、栖息、天敌、繁殖、和其他物种的关系。不是"游戏里该怎么设计"——是"自然界里怎么发生的"。
2. **我们的标签集合能不能表达这个？** 缺标签就补。不扭曲自然行为来适配现有标签。
3. **有没有当前行为违背了第 1 条？** 窄门（必须踏入同格）、魔法数字（不是从标签推导的数值）、画蛇添足的补丁（狼不需要学狗驱赶）——这些都是"用代码补设计"的信号。

**纪律**：如果一个机制需要给自然行为加"驱赶"类补丁，说明自然行为本身没写对。反向追溯根因，不补代码。

### 3.2 新卡/新规则（每次必查）
- [ ] CardDB 标签完整 + CARD_CAPABILITIES 登记
- [ ] WorldRules 有 is_xxx 判别函数 + 域名映射
- [ ] ecosystem_behavior_key 注册 + behavior 脚本存在
- [ ] _CARE_CHILD_TYPES 登记（如有亲子关系）
- [ ] population_manager 生命周期
- [ ] CardRuleAudit 六维归类
- [ ] 标签机合规：behavior 脚本无 card_type ==，规则查询在 WorldRules 内

### 3.3 可视化（每次必查）
**原则**：万物皆卡，万物皆可查。任何卡、任何格子——如果它有设定、有容纳、有状态，选中后必须全部可见。不在游戏里隐形。
- [ ] **外观 UI**：新卡在地图上有视觉呈现。不可见的机制卡（den/humus）必须有地面标记，被容纳的卡（窝内/藏有）有入口标记
- [ ] **选中 UI**：点击卡或空格子，信息栏展示全部标签（中文翻译）、能力、容纳内容、状态、HP
- [ ] **零遗漏**：不存在"这块没做 UI"——每个卡类型和格子类型都有信息栏渲染

### 3.4 参数调优（非架构改动，简查）
- [ ] 常量改在 game_state.gd
- [ ] 数值总表已同步

## 5. 工作纪律

### 5.1 每轮开始
1. 读 `work.log` 最后 30 行。
2. 检查未回复的 handoff。
3. 检查孤儿 WORK_START → 报告用户。

### 5.2 每轮结束（work.log 追加）
```
### 时间戳 | agent名 | PROMPT
### 时间戳 | agent名 | WORK_START
### 时间戳 | agent名 | WORK_END
```

### 5.3 修 bug 时

**DeepSeek**：只写现象（什么表现）+ 根因（哪个环节出了问题），**不指定具体改法**。修法由 Cursor 自行判断。

**Cursor**：修 bug → fix-log.md（只记事实）→ work.log 一行 `BUGFIX`。

### 5.4 审查结果（DeepSeek → work.log）
```
### 时间戳 | deepseek-v4 | REVIEW
结果: PASS / FIXES_NEEDED (N 项)
```

### 5.5 任务类型区分

| 类型 | 流程 |
|------|------|
| 参数调优 | 直接 handoff |
| 新卡/新规则 | 设计文档 → handoff → Cursor → 审查 → F5 |
| Bug 修复 | DeepSeek 写分析（现象+根因，**不写具体改法**），Cursor 自行判断修法并记 fix-log |
| 架构改造 | **架构命题 → Cursor 审查 → Cursor 拆执行计划 → DeepSeek 最终审查** |
| 架构审计 | DeepSeek 直接做，不通过 Cursor |

### 5.6 Cursor Composer 模式建议

DeepSeek 在每次交给 Cursor 的任务末尾注明建议模式。规则：

| 任务性质 | 模式 | 原因 |
|---------|------|------|
| 架构命题审查（读多文件、分析依赖链） | **Max Mode** | 需要大上下文 + 深度推理 |
| 写执行 handoff（自己拆方案） | **Standard** | 异步，人不等待，成本低 |
| 实现 handoff（逐条改代码） | **Fast** | 人在等，低延迟优先 |
| 大规模重构（跨 5+ 文件、改数百行） | **Max Mode** | 需要完整代码上下文 |
| Bug 排查+修复 | **Fast** | 交互式来回 |
| 代码审查 | **Standard** | 异步审查，延迟无关 |
| 写测试/补文档 | **Fast** | 范围明确，快速反馈 |

判断原则：需要理解全局依赖 → Max；人在等交互 → Fast；异步后台 → Standard。

## 6. 常规 handoff 模板

**原则**：不写语义描述（"应该看起来像XX"），只写精确修改点（"文件、函数、改成什么"）。Cursor 是执行引擎，不是设计者。

### 模板

```markdown
# 标题

## 1. 文件：`path/to/file.gd`

**函数**：`func_name()`

**现状**：<一句话说当前行为>

**改为**：
<精确改法——代码、逻辑、条件>

## 2. 文件：`path/to/file2.gd`

...

## 验收

- [ ] 具体可测的条件
- [ ] L0 断言数 ≥ N
```

### 规则

1. 每条改动 = 文件路径 + 函数名 + 现状 + 改为
2. 不写"应该""期望""看起来""感觉"——只写数据和逻辑
3. 验收条件必须可程序化验证，不写"玩家能看到"
4. 翻译/UI 文本 = 直接给目标中文，不写"按习惯翻译"

### 示例

```markdown
**函数**：`build_cell_lines(gx, gy)`

**现状**：无论单元格是否有 overlay，标题始终取 cell_display_title

**改为**：先查 overlay_card_at_cell(gx, gy)。若存在且 tag 含 "den"→ title = "狼窝"；tag 含 "fertile"→ title = "腐殖土"。无 overlay 时保持原逻辑。
```

## 7. 架构命题格式

架构命题文件命名：`proposition_<主题>_deepseek-v4.md`

### 7.1 命题结构

```markdown
# 架构命题：<一句话目标>

## 目标状态
<改完后的世界是什么样的——用行为来描述，不写代码>
例："新增一张带 capability.hunt 的卡，不改任何分发代码，自动捕猎"

## 当前断点
<指向具体文件和行号，说明断在哪里>
例：ecosystem_behavior_key() L2096-2117 是 card_type→字符串的硬编码链

## 方向建议
<怎么改，但不写具体实现>
例：把 ecosystem_behavior_key 从 card_type 查表改成 capability 组装

## 边界条件
<已知的坑、不能碰的东西、验收标准>
```

### 7.2 Cursor 回复格式

Cursor 在命题文件末尾追加 `## Cursor 架构审查` 段：

```markdown
## Cursor 架构审查
- 方案可行性：<可行/有阻碍/不可行>
- 改动范围：<涉及文件数、核心改动点>
- 遗漏风险：<命题可能没覆盖的依赖点>
- 建议的执行顺序：<如果可行，推荐拆几步>
```

Cursor 此阶段**不写代码**，只做架构审查。

## 8. 禁止事项

- 不修改 work.log 历史条目。
- 不删除对方的 handoff 文件。
- 不在 handoff 写超过 200 行正文（长内容放 docs/）。
- 不写入 API key、密码等敏感信息。
