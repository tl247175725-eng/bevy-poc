# 方寸商国：世界运转共识文档（v0.6）

> **正本：** `docs/design/core/世界运转共识_v0.6.md` · 入口 `docs/README.md`

## 1. 文档目标

本文件用于统一以下内容：

- 世界规则：什么一定会发生、什么绝对不会自动发生
- 行为规则：玩家卡如何按需求行动，避免"卡住 / 反复制造 / 躲到死"
- 修改流程：每次改代码前先对齐规则，再实现

本文件优先级高于临时口头设想。若与代码不一致，先更新本文件并确认，再改代码。

---

## 2. 核心设计原则（固定共识）

1. **需求驱动，不是脚本驱动**  
   玩家卡行动来源于"当前最高优先需求"，不是硬编码固定流程。

2. **物理一致性**  
   AI 与上帝视角遵循同一交互关系：走过去、搬运、叠加、砸击。  
   不允许"需求瞬移满足"。

3. **最小满足原则**  
   玩家卡只准备满足当前需求的最低物资，不做无限囤积。

4. **风险优先**  
   致命风险（狼近身、饥饿濒死）可打断非关键任务。

## 2A. 设计方法论：「先射箭，再画靶」

### 核心理念

游戏不是从"玩家应该干什么"出发设计，而是从"世界能不能自己运转到繁荣"出发验证。

### 第一阶段：射箭（让世界自己跑通）

让玩家卡 + 自动生态卡拥有足够的智能，能够利用世界的环境、资源、规则、其他自动卡，自主完成从"河边野地"到"桃花源商业帝国"的完整路径。

这一阶段玩家几乎不需要干预，世界自己可以达到理想终态。

目标：**验证"世界能不能自己运转到繁荣"。**

### 第二阶段：画靶（引入世界无法自解的问题）

分阶段加入新的事件、新的世界规则、新的限制条件。这些新变量会让玩家卡遇到自己无法解决的问题——例如狼群数量失控、旅人不再来了、木材枯竭、员工罢工。

这些"自动智能无法自解的关系问题"就是玩家需要面对的靶子。

### 第三阶段：玩家介入（这才是"玩"）

玩家从上帝视角，通过"手牌"系统介入世界。手中的每张牌都是一次改变世界关系的机会。

玩家的乐趣不是重复劳动，而是在：**观察 → 发现瓶颈 → 选择手牌 → 改变关系 → 观察新结果** 这个循环中体验"编排世界关系"。

### 关键判断

每次新增机制前，先问：

- 这是不是世界自己能处理的事？→ 如果是，应该由 AI 处理，不需要玩家手动做
- 这是不是 AI 无法自己解决的关系问题？→ 如果是，这才是玩家需要介入的决策点
- 这是不是在让玩家搬资源？→ 谨慎
- 还是在让玩家改变关系？→ 推进

## 2B. 统一交互语法：左右键 = 世界的唯一操作逻辑

### 核心原则

《方寸商国》中，所有卡牌与世界交互的方式只有两种：**左键**和**右键**。

这不是"玩家 UI 的两种操作"，而是**世界本身的两种基本语法**。

### 左键 = 物理作用

代表：砸、推、攻击、强制施力

特点：有方向、有阻挡、有节奏、有碰撞、可造成伤害/眩晕

规则：
- 两次有效接触触发结果（第 1 次红色数字 + 抖动，第 2 次触发结果）
- 持续接触不重复计数
- 左键拖拽松手后卡牌停留在目标位置

### 右键 = 关系作用

代表：叠加、依附、状态改变、组合、本质跃迁

特点：幽灵移动、无碰撞、无阻挡

规则：
- 同类卡叠加 → 数量增加
- 有关系卡叠加 → 本质变化
- 无关系卡叠加 → 弹到最近空格

### 铁律：AI 与玩家共享同一套语法

玩家卡 AI 的一切行为，必须通过模拟"左键"和"右键"操作来实现。不允许 AI 使用任何玩家无法使用的"快捷方式"。

| 操作 | 玩家怎么做 | AI 怎么做 |
|------|-----------|----------|
| 制作篝火 | 左键拖动树枝砸碎石 2 下 | AI 走过去，拾取树枝，移动到碎石旁，执行 2 次砸击 |
| 叠加草棚 | 右键拖动木头叠草皮 | AI 走过去，拾取木头，移动到草皮旁，执行右键叠加 |
| 砍树 | 左键拖动斧头碰树 2 下 | AI 走过去（携带斧头），移动到树旁，执行 2 次砍击 |
| 屠宰羊 | 左键拖动小刀碰羊尸体 2 下 | AI 走过去（携带小刀），移动到尸体旁，执行 2 次切割 |

### 这条铁律的意义

1. **直觉统一**：玩家看到 AI 做任何事，都能完全理解——因为 AI 用的就是玩家自己也能用的操作
2. **规则唯一**：新增任何机制，只需要定义"左键/右键规则"，AI 和玩家同时获得新能力
3. **玩家卡是化身，不是特权**：玩家卡在世界中使用的是和人类玩家完全相同的操作语法，没有任何魔法
4. **禁止 AI 后门**：不存在 `AI.makeFire()` → 篝火凭空出现，不存在 `AI.cook()` → 熟肉凭空出现

### 每个新机制必须回答

- 它的左键行为是什么？（两次接触后触发什么？）
- 它的右键行为是什么？（和什么叠加产生什么关系？）
- AI 能否通过模拟左右键操作来使用这个机制？
- 玩家看到 AI 做这个操作时，是否一眼能看懂？

## 2C. 手牌系统（预留记录）

### 定位

手牌是玩家从上帝视角介入世界的方式。它不是当前 MVP 阶段需要实现的内容，但应作为后续"画靶阶段"的核心交互。

### 初步设想

**手牌来源：**
- 从世界中拿取卡牌（将世界卡变为手牌）
- 通过规则获得（旅人成交、完成成就、季节轮替）

**手牌打出：**
- 将手牌中的卡牌放入世界中（直接放置）
- 通过手牌改变世界关系（关系介入、环境改变）
- 有资源约束和冷却限制，不能无限使用

### 当前状态

本系统仅为预留记录。当前阶段（射箭）应专注让玩家卡 + 自动生态卡自主跑通完整链条。

## 2D. AI 架构设计：「模块随世界进化」

### 核心原则

玩家卡 AI 的目的不是变得像真人一样"通用聪明"，而是**像真人一样行动**——在当前世界条件下，做出看起来像真人会做的决策。

因此 AI 能力不应该是固定在代码中的"预装智能"，而应该**随世界内容的增加而自然增长**：

```
世界有什么 → 哪些模块激活 → AI 懂什么
世界没有   → 模块不激活  → AI 不考虑

世界加入火车卡 → 火车模块激活 → AI 学会评估火车依赖链
世界加入火灾蔓延 → 火模块扩展 → AI 学会灭火和火场逃生
```

这不是"AI 变聪明了"，而是"世界的规则变多了，AI 的行为范围自然跟着变大"。

### 三层架构

```
┌──────────────────────────────────────┐
│         感知层 (Perception)           │
│  扫描世界：有什么卡、什么威胁、        │
│  什么生产链、什么依赖关系             │
│  （通用层，不随模块变化）              │
└──────────────┬───────────────────────┘
               ▼
┌──────────────────────────────────────┐
│         世界模型 (World Model)         │
│  对世界状态的唯一查询接口：            │
│  - 当前有哪些卡牌                     │
│  - 哪些依赖根存在/缺失                │
│  - 哪些威胁活跃                       │
│  （所有模块读同一数据源，不重复评估）   │
└──────┬───────────────┬───────────────┘
       ▼               ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  火模块   │    │  狼模块   │    │ 食物模块  │
│ 激活条件: │    │ 激活条件: │    │ 永远激活  │
│ 世界有火   │    │ 世界有狼   │    │          │
│ 木/石     │    │          │    │          │
│          │    │          │    │          │
│ 返回:    │    │ 返回:    │    │ 返回:    │
│ 依赖根=  │    │ 威胁=狼   │    │ 压力=饿  │
│ 篝火      │    │ 距2格     │    │ 85%     │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │
     ▼               ▼               ▼
┌──────────────────────────────────────┐
│      绝对前提分析器 (Root Analyzer)   │
│                                      │
│  收集所有激活模块的返回 →             │
│  按"不解决会导致的崩溃程度"排序 →     │
│  威胁（会死）> 依赖根缺失 > 压力       │
│                                      │
│  → 输出：当前唯一绝对前提              │
└──────────────────────────────────────┘
```

### 迷雾约束：感知层 ≠ 全图扫描

感知层的扫描范围严格受限于**迷雾系统**。玩家卡只能感知"已探索且当前可见"的卡牌和地形。这意味着：

**可见范围组成：**
- 玩家卡自身视野（周围 N 格）
- 已探索区域（迷雾已揭开、即使玩家卡不在场也保留记忆）
- 特殊光源/设施扩大视野（篝火、瞭望塔等）

**世界模型只保存可见信息：**
- 已探索区域的卡牌：保留在记忆中，但需标记"最后已知状态"
- 未探索区域的卡牌：不存在于世界模型中
- 离开视野后：未探索的区域重新被迷雾覆盖，卡牌信息丢失

**对 AI 行为的影响：**
- "篝火缺失"：只有在已探索区域内搜索不到篝火，才判定为缺失
- "狼威胁"：只有可见的狼才触发威胁评估
- "资源评估"：只基于已探索区域内的已知资源
- 玩家卡不知道自己看不见的东西

### 探索：一个底层需求

探索不是"闲着没事干"时的娱乐行为，而是一个**底层驱动力**。

**什么时候需要探索：**
- 已知区域内资源无法满足当前需求（如：可见区域没有石头，但需要制作工具）
- 已知区域内缺少关键依赖根（如：没有篝火、没有草棚材料）
- 观察到信号（如：动物迁移方向暗示远方有水源）

**探索行为：**
- 向未探索的格子移动
- 优先探索"河岸方向"（河流沿线资源更密集）
- 在探索过程中保持基础避险（不冲进已知的狼窝方向）

**探索与绝对前提的优先级关系：**
- 如果已知区域内没有条件满足绝对前提 → 探索提升为依赖根级
- 如果存在已知路径可以满足绝对前提 → 先满足前提，再探索

### 初始可见区域

游戏开局时，玩家卡周围有一块**已探索的初始区域**，其中包含满足生存基本前提的物资（碎石、树枝等），保证玩家卡能在一个可见范围内完成初始的绝对前提链（如生火）。

这避免了"开局一片黑，玩家卡不知道往哪走"的问题。

### 模块定义

一个模块是一段独立的行为逻辑，绑定到特定世界内容的出现：

```
模块 = {
  激活条件:     "世界存在 X 类型卡牌或 X 规则已启用",
  关心的依赖根:  [篝火, 草棚, ...],   // 缺失时汇报
  关心的威胁:    [狼靠近, 火蔓延, ...], // 活跃时汇报紧急程度
  关心的压力:    [饥饿值 > N, ...],   // 低于威胁和依赖根
  行为函数:     executeModuleAction()  // 通过左右键模拟实现
}
```

### 关键规则

1. **模块不互相通信**。所有模块通过"世界模型"读写世界状态，不直接互相调用。
2. **模块不"规划"**。每个模块只报告"我的领域有什么问题"，不独自决定"现在该做什么"。
3. **绝对前提分析器是唯一决策点**。所有模块返回的信息汇集后，由分析器做最终判断。
4. **新增世界内容 = 新增或扩展模块**。不需要修改分析器或感知层。
5. **AI 不会"超前聪明"**。世界没有的内容，对应模块不激活，AI 不产生无用计算和违和行为。

### 与传统优先级系统的区别

| | 传统固定优先级 | 模块化绝对前提 |
|---|---|---|
| 判断依据 | "安全比饱腹重要"（硬编码） | "当前情境下，不解决 X 会导致最大级联崩溃" |
| 同需求内部 | 看不出子节点依赖 | 能识别"火车头 > 车厢"、"篝火 > 烹饪" |
| 情境敏感度 | 低（优先级基本固定） | 高（火灾中灭火，灭火失败后切换逃跑） |
| 扩展方式 | 修改优先级列表 | 新增模块，不碰原有模块 |
| AI 表现 | "笨死"（困在长链中） | "果断"（打断低价值任务，先解决威胁/依赖根） |

## 3. 需求评估模型（绝对前提模型）

### 评估流程

每帧执行一次评估：

```
Step 1: 感知层扫描世界 → 更新世界模型
Step 2: 遍历所有激活模块，收集：
        - 紧急威胁（会死）
        - 依赖根缺失（级联崩溃）
        - 需求压力（会恶化）
Step 3: 绝对前提分析器排序：
        威胁 > 依赖根缺失 > 需求压力 > 无所事事
Step 4: 输出当前唯一主需求
Step 5: 从对应模块获取行为函数，通过左右键任务状态机执行
```

### 绝对前提的两种形态

**生存型绝对前提（动态、情境切换）：**

```
火势失控：
  当前绝对前提 = 灭火
  灭火失败 → 重新评估后果链
  → 营地保不住了，留在原地会死
  → 绝对前提切换为：逃跑

狼逼近：
  当前绝对前提 = 逃向篝火
  如果篝火不存在 → 跑向草棚/边缘/草丛
  
途中发现浆果 → 不中断（逃跑是绝对前提，吃不是）
```

**结构型绝对前提（依赖链分析）：**

```
火车贸易机会出现：
  依赖链分析：
    火车头 ─┬→ 车厢（有了车头才能拉车厢）
            └→ 铁轨（有了车头才能上轨）
  绝对前提 = 火车头（被其他节点依赖但不依赖链内其他节点）
  在火车头完成前，所有车厢和铁轨的投入都是浪费。
  
注意：
  如果火车头需要"锻造炉"才能制造，而锻造炉需要篝火升级……
  那么依赖链会追溯到：篝火 → 锻造炉 → 火车头
  绝对前提 = 篝火（最深的依赖根）
```

### 需求分类（替代原固定优先级列表）

以下为当前世界已激活模块的需求根，由绝对前提分析器动态排序：

**威胁级（强行抢占一切）：**
- 狼近身（无火庇护时）
- 火势蔓延至营地
- 饥饿濒死（倒计时中）
- 连续受击（反击或逃跑）

**依赖根级（不完成则后续无意义）：**
- 篝火缺失（火模块 + 狼模块 + 食物模块共同依赖）
- 草棚缺失（居住模块 + 火模块"火边草棚"共同依赖）
- 生产链根节点缺失（如火车头对火车链）

**压力级（不致命但会恶化）：**
- 饥饿值 > 阈值
- 工具缺失
- 营地库存不足

**安稳级（生存稳定后放行）：**
- 经营服务
- 营地整理
- 无所事事 / 夜归

## 4. 世界规则（当前共识）

## 4.1 河流规则

- 落在 `river/ford` 的卡会自动弹到邻近陆地（无需手动再拖一次）。
- 例外仅限明确可驻水规则的对象（后续扩展时单独列入）。

## 4.2 场域规则（三域同尺寸）

三种可视场域范围一致（与篝火一致）：

- 热场域（篝火）：橙色
- 储存场域（草棚）：蓝色
- 经营场域（桌子）：金色

说明：场域是"影响范围可视化"，不是"自动替玩家搬运"的代理逻辑。

## 4.3 篝火放置/清场规则

- 篝火不能生成在水属性不安全位置。
- 篝火**不会自动把可搬运卡全部弹走**。
- 篝火**不会自动自己挪位**处理障碍；由玩家卡基于需求判断后行动。

---

## 5. 玩家卡任务生命周期（防死循环）

每个任务统一使用状态机：

`plan -> move -> pickup -> move -> drop -> act -> done/fail`

统一中止规则：

1. 若正在 `pickup/move-with-carry`，禁止直接取消任务（避免"拿起-放下-重拿"死循环）。
2. 遇紧急危险可中止，但需先落地当前携带物，再切换逃生任务。
3. 任务失败必须记录失败原因并冷却短时间，避免下一帧立刻同条件重试。
4. 同一时刻只允许一个主任务 `craftTask`。

## 6. 制作与战斗共识

## 6.1 配方方向

- 除非明确写"必须先后"的关系，否则默认双向可叠加。  
  示例：`树枝 + 碎石 = 长矛`（双向）

## 6.2 尖锐物屠宰差异

- 小刀：最佳屠宰（满收益）
- 其他尖锐物（斧/矛/碎石等）：有损耗概率（示例：50% 几率少 1 肉）

该规则应在手动与 AI 两条路径保持一致。

## 7. 经营循环共识

- 旅人应形成"来-消费/未满足-离开-再来"的循环。
- 玩家卡在经营需求下应主动把旅人需求商品搬到桌旁。
- 经营行为不得破坏更高优先需求（安全、饱腹、前提生存）。
- 经营放行门槛（生存稳定）：
  - 饱腹值 `< 50`
  - 营地内至少有 2 种任意食物储备
  - 必须存在篝火
  - 以上条件连续满足 5 秒

## 12. 全局共性重构（v0.3 新增）

以下 6 条为后续工程改造的统一硬规则，优先级高于局部修补逻辑：

1. **统一决策内核（Need Engine）**  
   所有玩家卡行为必须先经过同一个需求评估入口：  
   `evaluateNeedState() -> 产出唯一主需求 + 目标 + 执行计划`。  
   禁止在多个函数中散落地直接触发行动。

2. **统一任务状态机（Task FSM）**  
   所有任务必须遵循同一生命周期：  
   `plan -> move -> pickup -> move -> drop -> act -> done/fail`。  
   不允许新任务绕过该状态机。

3. **统一配方引擎（Recipe Engine）**  
   手动操作、AI 自动执行、拖拽逻辑都调用同一配方查询与验证：  
   - 先验关系是否存在  
   - 再执行动作  
   - 错误时返回纠偏路径  
   禁止"某一条路径可合成，另一条路径不可合成"的分裂行为。

4. **统一资源复用策略（Existing First）**  
   所有需求先检查并复用地图已有成果，再考虑新建。  
   例如：先搬已有草棚到火域，再建新草棚；先取现有工具，再制作工具。

5. **统一放行门槛与抢占规则（Gate + Preemption）**  
   经营、夜归、避险、反击等触发条件必须走统一门槛判断，  
   且高优先级需求强制抢占低优先级任务。

6. **统一失败记忆（Failure Memory）**  
   同一失败原因应记录并短冷却，防止下一帧重复同错动作，  
   例如反复拿错材料、反复在无配方关系上尝试叠加。

## 16. 稳定性验收结论（v0.3.2）

验收范围：

- 长时段快照：`120s`（2.5 倍速）
- 连续抽样：`3 x 30s`（2.5 倍速）
- 死亡路径回归：`player-death-test`（最长 210s）

结果结论：

1. **稳定性通过（引擎层）**  
   未出现脚本报错、页面崩溃、任务状态机失控；玩家可进入"尸体态"并持续被世界规则处理。

2. **行为目标未通过（设计层）**  
   玩家主行为仍长期困在"生火前置碎石链"，最终常因饥饿倒下；未达到"生存→发展→经营"的目标链路。

3. **阶段判定**  
   当前版本适合继续做"生存前置链路打通"迭代，**暂不建议进入下一开发阶段（v0.4 功能扩展）**。

下一轮唯一优先项：

- 专项修复"生火前置碎石链阻塞"（含：石头获取后实际落地与可用性判定、携带石头阶段可达性、生火任务与饥饿抢占协同）。

补充诊断（v0.3.3）：

- 新增任务诊断埋点后发现：阻塞主因并非地形不可达，而是 `pickUpCard()` 未返回布尔值，导致任务层把"成功拾取"误判为失败，形成 `pickup_stone_fail` 风暴。
- 修复 `pickUpCard()` 返回值后，120s 诊断从 `makeShardForFire::pickup_stone_fail x70` 变为：
  - `makeFire` 完成 1 次
  - 主失败项转为 `preempted_by_higher_need`（需求抢占）
- 结论：本阻塞项第一归因从"世界结构不满足"修正为"需求实现层接口契约错误（函数返回语义）"。
- 继续调参后，`preempted_by_higher_need` 失败数量下降（由 4 降至 2），说明"远距安全威胁不打断原子动作"策略有效。
- 烹饪链补充"生肉目标重选"后，`missing_target_meat` 不再主导；失败收敛到 `missing_raw_meat`（世界里真实无可用生肉）。
- 已实现夜间睡眠规则：草棚旁休息时饥饿增速降为 1/3，且前 6 小时饥饿增量上限 20；每日睡眠不足 4 小时扣 2 HP。
- 已实现"主动巡猎 + 条件伏击"规则：取消篝火前纯等待猎物；仅在"附近有羊且有草卡"时进入草丛伏击，伏击冲出可连击 2 次。

## 第17节：工程修改原则——"先问结构，后写补丁"

**写入版本**：v0.3.4（2026-05-18）

### 原则

每当玩家卡（或任何 AI 实体）出现行为异常时，**第一个问题不是"怎么修这个 bug"，而是：**

> 这个问题在目前的框架下，有没有更本质、更干净的执行逻辑？

### 背景案例

本次修复"玩家有矛但在羊面前来回移动不攻击"时，最初走了"亡羊补牢"路线：
- 发现伏击阻断了直接攻击 → 加绕过判断
- 发现路径执行中无法攻击 → 在路径前加"攻击中断补丁"
- 发现狼逃打断了cookMeat → 加cookMeat豁免
- 发现伏击状态没被重置 → 加resetAmbush调用
- 每一个补丁制造下一个问题

**根本原因**：`updatePlayerNeeds` 把"反应性行为"和"规划行为"混在同一执行流里，`if(path.length) return` 这一行把所有反应性判断都挡在了路径执行之后。

**干净的解法**：将函数重构为三层分离：

```
[反应层] 每帧必执行（不受路径状态影响）
    R1: 狼威胁 → playerWolfReact(p)
    R2: 猎物在射程内 → 立即攻击
        ↓
[路径执行层] if(path) return
        ↓
[规划层] 评估需求 → 执行任务 → 执行需求计划
```

### 工程修改守则

1. **先描述问题的结构位置**：这个 bug 属于"反应层"、"规划层"还是"任务 FSM"的问题？
2. **问：现有架构是否支持干净解法？**如果需要在 5 处各加一个特判，说明架构层面出了问题，应先重构架构。
3. **补丁是临时的，结构是永久的**：只有在紧急修复或验证思路时才允许打补丁，确认后必须用结构性方案替换。
4. **每次重构后跑 turbo 测试**：确认行为链条（篹火→草棚→工具→狩猎→烹饪→存活）端到端通过，再向用户报告。
