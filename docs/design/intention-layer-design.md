# 意图层设计

> 讨论稿，不含实现细节。2026-06-03

---

## 一、问题

动物不需要意图层。饿→猎→吃。它的行为就是它的全部。

人不一样。人手里拿着石头同时饿了，可以做的事很多：猎兔、敲碎石制刀、收集树枝生火、烤肉。所有这些都在他的能力标签范围内。但先做哪个？**动物不需要选，人选不了。**

意图层回答的就是这个问题。不是"人能做什么"——标签已经回答了——而是"人现在想做什么，为什么"。

---

## 二、DF 为什么不行

矮人要塞的需求系统本质是数字排队。每个需求有数值，数值最高的排前面。矮人没有"意图"——它只是按照优先级执行任务列表，看起来像在做决定。

问题出在：数字之间没有互相理解。一个矮人可以同时"想喝酒 +10"和"怕被象踩 +8"，结果选了喝酒，被大象踩死。系统不知道"怕被象踩"应该无条件压制"想喝酒"。这不是 bug——这就是数字排队模型的上限。

---

## 三、我们的可能性：需求不是数字，是标签

`fire_bond` 不是一个"需要火的数值 = 40"。它是一个标签——意思不是"有多想靠近火"，而是"此生物必须保持火在燃烧"。像狼的 `care_child` 标签不是说"有多爱幼崽"，而是"如果窝里有幼崽没吃饱，停止其他行为去喂它"。**标签定义的是义务，不是渴望。**

那么人的需求标签怎么变成意图？

```
fire_bond：火未点燃 → 此项"不满足"
opportunistic：地上有可利用的原料 → 此项"存在机会"
饥饿：hunger > 55 → 此项"需要食物"
```

三项同时为真。意图引擎不是给它们打分排序，而是做反向规划：**哪个问题能解决？**

```
火未点燃 → 需要：干草 + 树枝 → 干草在火边等变干（被动），树枝在地上可捡 → 可行
需要食物 → 需要：猎物 OR 浆果 → 田鼠在灌丛里可猎但成功率低 → 可行但不优先
存在机会 → 石头在地上 → 可以制刀 → 制刀需要火先点燃 → 从属于火
```

引擎输出：

```
意图：生火 → 捡树枝
因为：火未点燃，树枝可捡，干草还在晒
```

玩家看到的是**一行意图 + 一行原因**。意图在变，原因在变，但它们永远是联动的。

## 四、核心机制：反向规划链

每个需求标签对应一个"满足条件"的检查函数。不是行为列表——是条件检查。

```
fire_bond → 满足条件：camp.anchor 格上存在已点燃的火
          → 若不满足：找出最近的路径
              → 缺树枝？→ 意图：捡树枝
              → 缺干草？→ 意图：等干草（火边放活草）
              → 有树枝+干草但无火？→ 意图：生火
              → 全满足？→ 此需求已满足，不产生意图

opportunistic → 满足条件：近期已完成一次材料转化
             → 材料转化列表从哪里来？从环境读——地上有什么、能拿什么
        
饥饿 → 满足条件：hunger < 30
      → 若不满足：找可食物。
          有矛 → 可猎（猎什么？跑 is_hunt_target_for）
          无矛 → 可采浆果/蘑菇
          有尸肉 → 可清腐取肉
```

每个需求独立运行自己的反向规划链，产出一个"候选意图"。多个候选意图产生后，意图引擎做**唯一选择**。选择原则：有硬依赖的意图优先于没有的。

"生火需要树枝"→ 硬依赖。必须做。
"饿了可以猎兔"→ 软意图。可以推迟。

## 五、展示层

信息栏不再显示"状态：求生 / 目标：得先把火生起来"。改为两个动态字段：

```
意图：捡树枝
关注：火未点燃 · 饿了 · 地上有石头
```

意图是当前正在执行的行动方向。关注是驱动决策的环境因素——让玩家看到"他在想什么"。

当意图改变时，原因是透明的——不是因为某个隐藏数值变了，而是关注列表里的某一项解析出了可行路径。

---

## 六、和标签系统的关系

意图引擎不替代 WorldRules，而是在 WorldRules 之外新建一层：

```
标签层 → 定义能力边界（已有）
行为管线层 → 消费能力标签，执行动作（已有）
意图层 → 消费需求标签 + 世界状态，产出短期目标（新增）
展示层 → 展示意图+关注（新增）
```

动物不走意图层。`is_ecology_starvation_subject` 为真的卡走行为管线，意图层只服务于人卡。

## 七、开放的坑

**需求标签到底有哪些？** `fire_bond`、`opportunistic`、饥饿——这三个是已有的。人的需求显然不止这些。安全需求？社交需求？娱乐需求？这些怎么定义？

**硬依赖怎么判断？** "生火需要树枝"——这是规则引擎怎么知道的？是写在规则里的（树枝 = fire 的 fuel 材料），还是硬编码的？如果是标签驱动的（material.lumber + fuel 标签 → 燃料），那就优雅。如果不是，就回到了脚本。

**候选意图之间的冲突？** 如果"捡树枝"和"躲狼"同时为真——躲狼硬压制，因为它读的是世界状态不是需求标签。威胁检测永远压制意图。

**什么情况下人什么都不做？** 全满足 + 无威胁 → idle。这是人的发呆状态，也是信息栏最简单的状态。
