# 世界规则标签机 v0.1

本文件定义“世界规则标签机 / 卡牌封装”的工程目标。它不是一次性重写规则引擎的方案，而是后续每次新增卡牌、关系、AI 任务时必须遵守的结构边界。

## 1. 目标

当前游戏的核心不是复杂物理模拟，而是“卡牌在世界规则中被识别、组合、执行”。因此长期目标是：

- 新增卡牌时，优先通过标签进入已有规则，而不是让每个任务脚本单独认识它。
- 修改规则时，能看出影响哪些卡牌、哪些任务、哪些玩家可见行为。
- 出现发呆或不兼容时，能归因到标签缺失、状态缺失、执行入口缺失、还是设计例外。
- 规则迁移必须小步可验收，禁止一次性改成完整抽象规则引擎。

## 2. 核心定义

**标签**描述卡牌稳定能力，不描述临时任务过程。

例如 `material.lumber`、`food.edible`、`camp.storable` 是标签；“正在去取水”“建棚阶段 3”不是标签。

**卡牌封装**由三部分组成：

- 静态定义：`CardDB` 中的类型、名称、层级、标签、商品值、脚本绑定。
- 运行时状态：位置、是否被携带、是否在窝内、是否眩晕、是否成熟等待、是否正在被任务占用。
- 可参与规则：右键关系、敲击配方、环境计时、AI 任务入口、生态行为入口。

**世界规则**至少包含：

- 输入选择器：按类型、标签或状态找到候选卡牌。
- 前提条件：距离、手持/地上、是否结构锁定、是否已有目标产物。
- 关系执行：右键合成、敲击配方、环境变化、生态动作。
- 执行入口：玩家任务、生态行为、手动交互、环境自动。

## 3. 工程约束

- 任务脚本应消费 `WorldRules` 或审计过的规则查询结果，避免散落卡名判断。
- 允许专名规则，但必须写明原因。例如长矛当前仍只使用 `twig`，不是所有 `material.lumber`。
- 每个核心链路都必须能回答：玩家是否能执行、手动是否能执行、生态是否能执行、失败后如何恢复。
- 同一张卡在不同状态下是否仍保留能力必须明确。例如 `stone` 在地上和手持时都应能参与制碎石。
- 审计表中的 `待定` 是设计债务，不是默认通过；进入主线前必须清掉。

## 4. 审计产物

- `docs/design/generated/card-rule-facts.md`：自动导出的代码事实，只记录 `CardDB`、relations、impact recipes。
- `docs/design/card-rule-audit.md`：全卡人工审计表，记录运行时状态、AI 可达性和风险结论。
- `scripts/test/card_rule_audit.gd`：一致性检查与导出逻辑，接入 L0 单元测试。

## 5. 下一步迁移原则

后续迁移按“发现问题 → 审计表确认 → 小切片代码迁移 → trace 验收”的顺序走。

优先迁移会造成发呆的问题：

- 手持状态没有进入规则查询。
- 手动可合成但 AI 不会触发。
- 标签过宽导致专名规则被吞掉。
- 标签过窄导致新卡不能进入已有任务。
- 等待/成熟/入窝等中间态没有被视为进展。
